Homepage

2 crit 3 high 3 med 1 polish

First viewport on a phone is the most-trafficked single screen on the site; this is where the cycle states have to land. Findings center on hero compression, the four CTAs that fight for the same tap zone, and a Beehiiv form that shows seams.

  1. Critical
    Two stacked CTAs in the hero have ~36px height — below tap target

    "Subscribe" and "View dashboard" buttons sit close together vertically; on 375px both compress and the gap between them is ~8px, which means mis-taps land on the wrong action.

    Single primary CTA in the hero (full-width, min-height: 48px, padding: 14px 22px); demote the second to a ghost link below. Stack gap 16px minimum.

    375 · 414
  2. Critical
    Beehiiv subscribe URL renders as a raw inline link

    The subscribe-forms.beehiiv.com/… URL appears as link text inside the subscribe section. On mobile it wraps mid-URL, breaking flow and reading as a developer leftover.

    Replace with the Form panel component (email input + primary button). Iframe the Beehiiv form behind it, or POST directly to their endpoint and hide the URL entirely.

    375 · 414
  3. High
    Hero headline likely scales below 28px on 375px

    The two-clause headline ("Track the Cycle, Seize the Trend — Turning Macro Insight into Market Clarity") is long enough that any responsive scaling will compress it past readable display size on 375px.

    Cut to one clause. Set hero with font-size: clamp(32px, 9vw, 56px); line-height: 1.05; text-wrap: balance; — never below 32px on 375.

    375
  4. High
    Four "what you get" tiles likely render as four narrow columns or a cramped grid

    Below the hero, four bold-prefixed paragraphs (Weekly research / 4-state indicator / Direct delivery / Allocation focus). On phone these tend to either shrink to half-width pairs (cramped) or stack with no rhythm.

    Stack to single column on <720px with gap: 24px; reduce to a three-line mono row above the next section, separated by " · " glyphs.

    375 · 414
  5. High
    Sample-issue cards likely have 12–14px body type on phone

    Three sample-issue blocks ("Sample Issue / Historical Lens / Framework Note") tend to inherit a desktop-tuned body size that's been globally scaled down. Mid-teen body on a small screen is fatiguing.

    Force body to 16px minimum on mobile via @media (max-width: 720px) { .card p { font-size: 16px; line-height: 1.55; } }.

    375 · 414
  6. Medium
    Cycle-state pills probably wrap to two lines on 375px

    "BTC Accumulation / Alt Rotation / Take Profit / Risk Off" rendered as four side-by-side pills won't fit horizontally on 375px without truncation or wrapping.

    On <720px, switch the rail to a 2×2 grid (grid-template-columns: 1fr 1fr) with the active state visually distinct, not pills. Or single-column dot list, active state larger.

    375
  7. Medium
    Page side padding likely 16px or less

    Default WordPress / common CSS templates set mobile gutters to 16px. On the dark editorial palette this feels claustrophobic; long lines run almost to the edge.

    Set padding-inline: 20px on mobile, with the inner container capped at max-width: 100%. Matches the design-system spacing scale (--s-7 desktop, --s-5 mobile).

    375 · 414
  8. Medium
    "Trust line" trio at footer likely floats as three orphan phrases

    "Research only · UK/US legal framing · No performance claims" — on phone these tend to wrap awkwardly with the centre dots stranded.

    Set as a single mono line (11px / 0.16em / uppercase) with text-wrap: balance; on 375 stack vertically with the dots removed.

    375 · 414
  9. Polish
    Italic display text at small sizes loses character

    If hero italics drop below ~22px on phone, Fraunces' optical-size mechanism may not engage and the italic character flattens.

    Use Fraunces' opsz axis explicitly: font-variation-settings: "opsz" 36; on the hero, even at smaller computed sizes — keeps display character at phone scale.

    375
/dashboard/

Dashboard

2 crit 3 high 2 med 1 polish

The Tape & Margin direction works because the two columns collapse cleanly. Anything sticky or position-fixed is the highest mobile risk — and the page's async loading is invisible if not handled deliberately.

  1. Critical
    Sticky tape rail likely doesn't release on mobile

    A 260px sticky left rail collapsed to a horizontal scroller on phone can fail in two ways: (a) position: sticky with no top reset misbehaves on iOS Safari momentum scroll; (b) the rail keeps its desktop width and forces horizontal page scroll.

    Wrap the media query: @media (max-width: 900px) { .tape { position: static; max-height: none; display: flex; gap: 12px; overflow-x: auto; -webkit-overflow-scrolling: touch; } }. Verify no parent has overflow: hidden that breaks sticky on desktop.

    375 · 414
  2. Critical
    Async load shows blank cycle state for ~600ms

    If "BTC Accumulation" hydrates from a fetch, mobile users on slow LTE will see an empty hero or layout shift. The page's whole point is the cycle state — invisible at first paint is unacceptable.

    SSR the last-known cycle state into the HTML at build/publish time as the default. Hydrate over it only if the value changed in the last hour. Wrap any fetch-dependent block in min-height equal to the loaded height to prevent CLS.

    375 · 414
  3. High
    Loading state likely shows nothing or a default browser spinner

    Async data without an in-system skeleton reads as a broken page on phones, where the user can't easily check the network tab.

    Add a skeleton variant of each card: same dimensions, hairline border, mono "loading…" eyebrow in --ink-3, no shimmer (calmer brand). Replace in place when data lands.

    375 · 414
  4. High
    Tape numbers likely don't use tabular-nums on mobile fallback fonts

    If JetBrains Mono fails to load on a slow connection, the system fallback is monospace but without tabular alignment — prices jitter when they update.

    Set font-variant-numeric: tabular-nums; on every numeric cell, regardless of family. Add font-display: swap; on the @font-face so the fallback renders immediately.

    375 · 414
  5. High
    Scorecard rows likely wrap awkwardly on 375px

    A row like "M2 108.4T ↑ supportive" with name + value + arrow + status on one line will wrap mid-row on 375 — the status pill stranded on its own line.

    On <480px, restructure each row as two lines: name + value (top), arrow + status (bottom, mono 11px). Hairline rule between rows unchanged.

    375
  6. Medium
    "Refreshed Nm ago" timestamp probably wraps under the brand

    Mast meta with brand on the left and timestamp on the right collapses to two stacked lines; the timestamp ending up under the wordmark looks like a subtitle.

    On <720px, right-align timestamp to its own row, mono 11px ink-3, separated from the brand by a hairline rule.

    375 · 414
  7. Medium
    Editorial column on the right likely runs full-width with no measure cap

    When the two-column collapses, the right column inherits the page width — so editorial paragraphs can run to ~50ch which is fine, but on landscape phone or 414 they may push past 70ch where readability drops.

    Wrap article body in max-width: 62ch; margin-inline: auto; even on mobile. Keep the tape and CTA full-width.

    414
  8. Polish
    Live-dot pulse animation on slow phones can feel jittery

    A box-shadow keyframe pulse on the orange dot (if implemented) is GPU-cheap on desktop but on older Android can stutter at 30fps.

    Use opacity/transform: scale() rather than animating box-shadow. Add will-change: opacity, transform; only while pulsing. Disable under @media (prefers-reduced-motion: reduce).

    375 · 414
/newsletter/

Newsletter

1 crit 2 high 2 med 1 polish

This is the conversion page on phones — the input field is the page's whole point. Beehiiv's default styling is the consistent failure mode across small sites that use it.

  1. Critical
    Beehiiv iframe likely overflows 375px viewport

    Embedded subscribe iframes default to ~500px width and don't respect the parent. On 375px this triggers horizontal page scroll — the most diagnostic mobile failure there is.

    Wrap iframe in a container with width: 100%; max-width: 100%; overflow: hidden; and set iframe { width: 100%; }. If Beehiiv won't reflow, replace with a custom form POSTing to their endpoint.

    375 · 414
  2. High
    Email input likely uses type="text" not type="email"

    Default Beehiiv embeds sometimes ship as text — wrong iOS keyboard (no @ key surfaced), no email validation hint.

    Force type="email" inputmode="email" autocomplete="email" on whatever input is rendered. If you can't reach the input, replace the iframe with a 3-line custom form.

    375 · 414
  3. High
    Subscribe button likely too narrow / not full-width on mobile

    Inline-with-input button works on desktop, fails on phone — either input shrinks to ~200px or button gets cut off.

    On <720px: stack input above button, both width: 100%, button min-height: 48px, gap 10px.

    375 · 414
  4. Medium
    Sample-issue cards likely repeat homepage and feel duplicative

    Even outside layout terms — on a phone, hitting newsletter from the homepage and seeing the same three cards is more visible (no horizontal context to differentiate).

    Make this page the archive: 8–12 issues as a single dateline list (mono date, Fraunces 17px headline, 14px italic regime tag). Hairline rule between rows. Reuses no homepage components.

    375 · 414
  5. Medium
    "What you get" three-column block likely two narrow columns or a stack with weak rhythm

    Three columns of cadence/length/anchor will collapse on phone; default stacking has no separator.

    Stack with hairline rules between rows; mono eyebrow stays orange-2; padding-block: 20px per row. Reads as a definition list, not a card grid.

    375 · 414
  6. Polish
    Success state after submit likely is the Beehiiv default green tick

    Off-brand, breaks the editorial frame at the conversion moment.

    Replace iframe success with your own confirmation: italic Fraunces "Welcome aboard. Your first issue arrives Tuesday." in the same panel chassis. Listen for the form's success postMessage if iframed.

    375 · 414
/methodology/

Methodology

0 crit 2 high 2 med 1 polish

Long-form pages tend to mobile better than marketing pages — fewer components to break — but reading ergonomics need attention.

  1. High
    No visible TOC on mobile makes the page un-skimmable

    A long page on phone with no contents block forces scroll-hunting; bounce risk.

    Add a TOC card at the top: bordered, border-radius: 12px, mono "CONTENTS" label, ordered list with anchor links. On phone keep it open — no collapse.

    375 · 414
  2. High
    Body type likely 16px Inter, not 17–18px Fraunces

    Methodology read on phone benefits from larger, slower-paced serif. Inter at 16 reads as documentation, not research.

    Set body font: 400 17px/1.6 var(--serif); text-wrap: pretty;. Keep measure at max-width: 62ch via auto margin.

    375 · 414
  3. Medium
    Indicator definition lists may use default <ul> bullets

    Default disc bullets on dark theme are ugly and hang outside the measure on phone.

    Replace with list-style: none + numbered counters in mono orange-2 in a 28px gutter, matching the section-header pattern in the design system.

    375 · 414
  4. Medium
    Inline code / variable names may break across lines on 375

    Long ticker symbols or code spans inside paragraphs without word-break rules cause horizontal overflow.

    Set code, .ticker { overflow-wrap: anywhere; } on the body scope.

    375
  5. Polish
    No "back to top" anchor on a long phone page

    After 4000+ pixels of scroll, returning to the TOC requires a long swipe.

    Quiet back-to-top link as a final element before footer — no fixed FAB. Mono "↑ Top" 11px, ink-3, centred.

    375 · 414
/contact/

Contact

0 crit 2 high 1 med 1 polish

Forms on phone live or die by input height, label clarity, and whether the keyboard hides the submit button.

  1. High
    Input fields likely < 44px tall

    Default input padding on most CSS resets gives ~36px input height; fails Apple's 44px touch-target guideline.

    Set input, textarea { min-height: 48px; padding: 12px 14px; font-size: 16px; }. The 16px size also prevents iOS auto-zoom on focus.

    375 · 414
  2. High
    Submit button gets pushed below the iOS keyboard

    When the user focuses the message field, the soft keyboard covers the submit button — they have to dismiss the keyboard to find it.

    Order fields so name → email → subject → message; submit appears immediately after the textarea. Add scroll-padding-bottom: 80px; on html so the focused field stays in view.

    375 · 414
  3. Medium
    Email link probably opens in mailto with no body template

    "Email us at info@" tap launches the iOS mail app with a blank message — friction-free but generic.

    Pre-fill subject: mailto:info@macropulse.uk?subject=Reader%20question. Adds zero work on either side and routes inbound mail.

    375 · 414
  4. Polish
    Form labels probably collapse to placeholders

    Placeholder-only labels disappear once typing starts; common accessibility complaint.

    Always show a mono 11px / 0.16em / orange-2 label above each input. Placeholder is supplementary ("e.g. you@firm.com").

    375 · 414
/faq/

FAQ

0 crit 1 high 2 med 0 polish

Accordion ergonomics on phone — expanding one question shouldn't push the others out of sight, and the tap zone has to be the whole row.

  1. High
    Default <summary> tap target is just the text

    Native disclosure widget makes only the text clickable; users tap white space and nothing happens.

    Set summary { display: block; padding: 16px 0; cursor: pointer; } so the full row is the target. Custom marker via summary::-webkit-details-marker { display: none; } + a styled "+/−" on the right edge.

    375 · 414
  2. Medium
    Opening one accordion may scroll the next one off-screen

    Browsers don't auto-scroll on <details> open; on phone the user opens question 3 and finds question 2's body now obscures it.

    Add a toggle JS handler that scrolls the opened summary into view minus a 16px top offset. Five lines of vanilla JS.

    375 · 414
  3. Medium
    No grouping makes the page a long flat scroll

    A flat list of 12+ Qs on phone is the same UX whether you scroll or use Cmd-F (which most phone users won't).

    Group into 3 sections (Newsletter / Cycle indicator / Subscriptions) with section-header components above each. No per-section accordion-of-accordions; just flat lists per group.

    375 · 414

About

Editorial page; lowest mobile risk. Issues centre on portrait sizing and bio measure.

  1. High

    Two-column desktop layout with portrait left + bio right collapses to portrait above bio; default behaviour fills the viewport with a single image, dominating the first scroll.

    Cap portrait at max-width: 200px; aspect-ratio: 1; object-fit: cover; border-radius: 12px; on phone. Keep it left-aligned, not centred.

    375 · 414

    414

    Identical CTA strip on multiple pages reads as scaffolding on phone, where it appears more often per session.

site-wide

Navigation & chrome

1 crit 1 high 1 med 0 polish

Findings that apply across every page — the hamburger, the nav drawer, the footer.

  1. Critical
    Hamburger likely lacks aria-expanded + aria-controls

    Default WordPress hamburgers ship with onClick that toggles a CSS class but no ARIA state. Screen readers and keyboard users can't tell whether the menu is open.

    On the button: aria-expanded="false" aria-controls="primary-nav"; toggle aria-expanded in the click handler. Set aria-label="Open menu" / "Close menu" to swap.

    375 · 414
  2. High
    Open mobile drawer probably doesn't trap focus

    When the drawer opens, Tab still moves through the page underneath; visually-hidden links remain reachable.

    When open, set inert on main and the rest of the body; focus the first link in the drawer; Esc closes.

    375 · 414
  3. Medium
    Nav links inside drawer likely < 44px tall

    Default link rows in mobile drawers often inherit body line-height (~24px); the visible link is ~24px tall but the surrounding padding gives it apparent size.

    Make each a in the drawer display: block; padding: 16px 24px; font-size: 18px;. That's 50px tap height with breathing room.

    375 · 414
